# qa 技术方案 -- 发布流程

> 需求: deploy-process | 角色: qa

## 测试策略

发布流程的测试以手动验证为主，因为涉及 SSH 连接、Docker 构建、远程服务器等外部依赖。

## 测试范围

| 测试项 | 验证内容 | 类型 |
|--------|---------|------|
| Docker 构建 | Dockerfile.backend 和 Dockerfile.frontend 能成功构建 | 手动 |
| docker-compose 启动 | 双服务正常启动，容器健康 | 手动 |
| nginx 反向代理 | /api/ 请求正确代理到后端 | 手动 |
| SPA 路由 | 前端路由刷新页面不 404 | 手动 |
| 端口检测 | 占用端口时能自动选择可用端口 | 手动 |
| 部署脚本 | deploy.sh 完整流程执行成功 | 手动 |
| 多环境 | 指定不同环境名执行部署 | 手动 |
| Web 端连接 | 浏览器访问部署地址可正常使用 | 手动 |
| 数据持久化 | 重启容器后数据不丢失 | 手动 |
| JWT 环境变量 | 生产环境使用注入的 JWT_SECRET | 手动 |

## 验收测试清单

1. 在项目根目录执行 `./deploy/deploy.sh`，观察完整部署流程
2. 部署完成后，浏览器访问输出的地址，确认页面正常加载
3. 注册/登录操作正常
4. 记一笔账，刷新页面数据仍在
5. 执行 `docker-compose restart`，数据不丢失
6. 修改 deploy.yaml 添加一个新环境，指定新环境部署
7. 在默认端口被占用的情况下重新部署，确认自动选择新端口

## 关键决策

手动测试优先，不编写自动化测试脚本。理由：部署流程涉及外部基础设施（SSH、Docker、远程服务器），自动化测试的维护成本高于收益。
