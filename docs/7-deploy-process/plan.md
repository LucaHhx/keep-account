# deploy-process

> 建立标准化的发布部署流程，支持一键部署到目标服务器，并自动管理端口与多端连接配置

## 目标

1. 开发者可以通过一条命令将应用部署到目标服务器
2. 支持灵活的多环境部署（不同服务器地址，域名或 IP）
3. 自动管理服务器端口，避免端口冲突
4. 部署后各端（Web/桌面/移动端）能正确连接到已部署的服务

## 范围

**包含:**

- 使用 docker-compose 编排前后端服务
- 部署脚本：支持通过 SSH 部署到远程服务器
- 多环境配置：支持添加多个部署目标（域名或 IP 地址）
- 端口管理：分析目标服务器端口占用，自动选择可用端口
- 客户端连接配置：部署后自动生成/更新各端的服务地址配置
- 默认部署目标：MINI1@192.168.0.228（SSH 端口 22，密钥 /Users/luca/.ssh/key）

**不包含:**

- CI/CD 自动化流水线（后续迭代）
- 蓝绿部署 / 滚动更新等高级发布策略
- 容器编排平台（Kubernetes 等）
- SSL 证书自动管理
- 服务器监控与报警
- 数据库迁移自动化

## 核心用户场景

- 场景 A：开发者完成功能开发后，执行一条部署命令，应用自动构建并部署到默认服务器（192.168.0.228），部署完成后终端输出服务访问地址
- 场景 B：开发者需要部署到一台新的服务器，通过配置文件添加新的部署目标（填写 IP/域名、SSH 端口、密钥路径等），然后指定该目标执行部署
- 场景 C：目标服务器的默认端口已被占用，部署脚本自动检测端口占用情况，选择一个可用端口进行部署，并在输出中告知实际使用的端口
- 场景 D：部署完成后，开发者在手机（移动端）、电脑浏览器（Web 端）、桌面应用（Tauri 端）分别打开应用，均能正确连接到已部署的后端服务

## 背景

当前项目已完成基础功能开发，需要建立标准化的发布流程。首个部署目标为局域网内的服务器（MINI1@192.168.0.228），后续可能扩展到其他服务器，地址可能是域名也可能是 IP。

项目由 Go 后端 + React 前端组成，后端使用 SQLite 数据库，前端通过 Vite 构建为静态文件。前端同时支持 Web、Tauri 桌面端和移动端，因此部署后需要确保各端都能正确找到并连接后端服务。

服务器上可能已有其他服务占用端口，因此需要在部署前自动检测端口可用性，选择未被占用的端口。

[备注] 用户明确要求使用 docker-compose 进行部署编排，具体的 Dockerfile 编写、构建流程、容器网络配置等技术细节由开发团队决定。

## 验收清单

- [ ] 执行一条命令可以完成应用的构建和部署
- [ ] 应用通过 docker-compose 在目标服务器上运行
- [ ] 可以通过配置文件管理多个部署目标（支持域名和 IP）
- [ ] 默认部署目标（MINI1@192.168.0.228）可以正常部署
- [ ] 部署前自动检测端口占用，选择可用端口
- [ ] 部署完成后输出实际的服务访问地址和端口
- [ ] Web 端可以通过浏览器访问已部署的服务
- [ ] 桌面端（Tauri）可以连接到已部署的后端服务
- [ ] 移动端可以连接到已部署的后端服务
- [ ] 添加新的部署目标后，可以指定该目标进行部署
